#### 具体实现细节
1. RRF = Σ 1 / (k + rank)  k=60 假如两路混合检索 一路第一名 一路第五名 那么他最终得分就是 1/61+1/65  

##### 目前已经实现(可能不全不建议作为参考)
1. BM25 + 向量检索 + RRF
- 流程如下：
文档入库时，系统会先保留原始 content，再用 jieba 对 title、source、tags、metadata 和 content 做中文分词，
得到一份更适合检索的分词文本；然后把这份文本写进 PostgreSQL 的 tsv，同时 提取一批 keywords 落库。
已有数据则通过回填脚本重建 keywords 和 tsv。                                                                                                             
                                                                                                                                                                                 
用户查询时，系统会先对问题做分词和关键词提取，过滤掉低价值口语词，再把高价值词拼成 OR 型 tsquery 去查 tsv，
得到关键词召回结果；同时还会走一条向量召回。最后用加权 RRF 融合两路结果，目前是 vector=1.0、keyword=0.5，
得到最终的 Hybrid Recall 排序。

2. rerank

部分信息：rerank前侯选数是20 最终输出chunk数是5

3. rewrite
1.用户问题先进入 retrieve                                                                                                    
   
  - 它不会先粗暴替换原问题，而是先保留原问题作为主查询。                                                                                        
2.做规则清洗                                                                                                               
  在 build_query_rewrite_result 里先做三件事：                                                                                
 - 统一空白和口语噪声                                                                                                                                                         
 - 修正常见错别字/简称/别名                                                                                                                                                   
 - 生成一条更偏关键词风格的 query                                                                                                                                             
3.按条件决定是否触发 LLM rewrite                                                                                               
只有在问题明显口语化、规则清洗变化较大、或者关键词太稀时，才会尝试让 LLM 额外生成 1 到 2 条检索 query。                                                                       
如果没配 DEEPSEEK_API_KEY、配置关闭、或者模型调用失败，会自动降级，只走规则层，不会把检索打挂。                                                                               
4.多路召回                                                                                                                
对每条 query 变体都会各自跑一遍：                                                                                                                                             
 - 向量召回                                                                                                                                                                   
 - BM25 关键词召回                                                                                                                                                            
 - 两路先做一次 RRF 融合     
 - 每一路都会得到40个chunk
5.全局融合
 - 再把“原问题 / 标准化问题 / 关键词问题 / LLM 改写问题”这些分支结果做第二次 RRF 融合。 
 - 权重上原问题最高，规则清洗次之，关键词和 LLM 改写更偏补充召回，不会反客为主。   
 - 最终得到20个排名最前的chunk
6.rerank 和 answer
 - 融合后的候选再按原问题做 rerank，最后问答仍然基于原问题生成答案，不会把回答意图完全交给 rewrite。
