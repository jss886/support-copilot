## 目标
搭建 multi-agent 分层协作架构，在已有 RAG 基础上把 orchestrator、retrieval、quality_gate、answer 等角色工程化落地，形成可编排、可扩展的 agent graph。

核心诉求：
- 不走"多 agent 平级自由讨论"路线，而是走"Graph + 局部 ReAct"的分层结构
- orchestrator 负责意图路由，retrieval 负责检索封装，quality_gate 负责质量兜底，answer 负责最终回答
- 支持 auto/direct/rag 三种执行模式，auto 模式下走 LLM 路由判断
- 在 RAG 链路中图片识别从 OCR 升级为 Gemini 多模态识别

### 已实现：

1. **Graph 编排层**（`supportAgents/graph/`）
   - `SupportAgentState` 统一共享状态（session_id、user_query、intent、route_reason、retrieval、quality、answer、error 等字段）
   - `create_initial_state()` 统一入口，支持 auto/direct/rag 三种模式
   - `build_support_graph()` 编译完整图：START → orchestrator → retrieval（或直接 answer）→ quality_gate → answer → END
   - `run_support_graph()` 作为 CLI/API 统一调用入口
   - `debug_graph.py` 调试脚本，可视化 graph 执行流程

2. **Agent 角色层**（`supportAgents/agents/`）
   - `orchestrator_agent`：LLM 驱动的意图路由，支持 doc_qa / code_qa / tool_only / direct_answer / fallback 五种意图分类
   - `retrieval_agent`：封装复用现有 rag/ 检索能力（query rewrite → hybrid retrieval → rerank），输出结构化 RetrievalPayload
   - `quality_gate`：检索质量检查节点，区分 passed / degraded_empty / degraded_low_score 三种质量标记，answer 节点按质量标记选择不同回答策略
   - `answer_agent`：接入 llm_clients 统一工厂，按意图选择不同 system prompt，综合上下文生成最终回答
   - `prompts.py`：集中管理所有 prompt（ORCHESTRATOR、ANSWER、TOOL_ONLY、FALLBACK、MEMORY），角色收紧为"企业内部技术支持 Copilot"

3. **LLM 客户端层**（`supportAgents/llm_clients/`）
   - `base_client.py`：客户端抽象基类
   - `deepseek_client.py`：DeepSeek 官方 OpenAI 兼容接口，支持 thinking 模式
   - `openai_client.py`：OpenAI 客户端
   - `anthropic_client.py`：Anthropic 客户端
   - `google_client.py`：Google Gemini 客户端
   - `factory.py`：统一工厂方法 create_llm_client()
   - `validators.py`：模型名校验

4. **三种执行模式**
   - `auto`：LLM 路由判断意图，doc_qa/code_qa 走检索链路，其他走直接回答
   - `direct`：跳过检索，直接走 answer 节点
   - `rag`：强制走完整检索链路（rewrite → hybrid retrieval → rerank → answer）

5. **Gemini 多模态图片识别**（`rag/image_caption.py`）
   - 替代原有 OCR 方案，对 RAG 链路中的图片使用 Gemini 多模态模型进行语义识别
   - 与文档 chunk 合并后一并写入索引
