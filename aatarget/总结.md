# 当前阶段总结

## 一、项目当前状态

这个项目现阶段已经完成了较完整的 RAG 基础能力，不是简单 demo。

已有能力包括：

- 文档入库、切片、向量化
- PostgreSQL + pgvector 检索
- 关键词检索 + 向量检索混合召回
- query rewrite
- rerank
- 基于上下文生成回答
- API 对外暴露
- retrieval / RAGAS 评测相关代码

核心已有模块主要在：

- `rag/`
- `api/`
- `cli/`

其中当前 multi-agent 相关的新结构放在：

- `supportAgents/agents`
- `supportAgents/graph`
- `supportAgents/dataflows`
- `supportAgents/llm_clients`

## 二、当前 multi-agent 设计方向

目前已经明确，不走“很多 agent 平级自由讨论”的路线，而是走更稳的分层结构：

- `llm_clients` 负责模型接入
- `agents` 负责角色能力
- `graph` 负责流程编排
- `dataflows` 负责外部数据接入

当前建议的第一版核心角色是：

1. `orchestrator_agent`
2. `retrieval_agent`
3. `action_agent`
4. `answer_agent`
5. `memory_agent`

整体协作关系是：

`orchestrator -> retrieval / action / direct / fallback -> answer -> memory`

当前优先实现的是最小闭环：

`orchestrator -> retrieval -> answer`

## 三、当前图设计结论

之前已经输出过一版初步 Mermaid 图，文件在：

- `aatarget/初步agents.md`

这版图的核心结论是：

- 外层适合用 Graph 做流程编排
- 不建议整个系统全链路都做 ReAct
- 更适合“Graph + 局部 ReAct”
- 第一版最适合把 ReAct 只放在后续 `action_agent`

## 四、llm_clients 当前进展

当前已经在 `supportAgents/llm_clients` 中补上了 DeepSeek 客户端：

- `supportAgents/llm_clients/deepseek_client.py`

并且已经接入：

- `factory.py`
- `validators.py`
- `__init__.py`

当前支持这样统一创建：

```python
from supportAgents.llm_clients import create_llm_client

client = create_llm_client("deepseek", "deepseek-v4-flash")
llm = client.get_llm()
```

当前实现特点：

- 默认走 DeepSeek 官方 OpenAI 兼容接口
- 默认 `base_url` 为 `https://api.deepseek.com`
- 支持 `thinking` 通过 `extra_body` 传入
- 已加入模型名校验

本阶段确认过的官方模型信息：

- 推荐模型：`deepseek-v4-flash`、`deepseek-v4-pro`
- 兼容别名：`deepseek-chat`、`deepseek-reasoner`

## 五、supportAgents 当前已补骨架

### 1. graph state

已新增：

- `supportAgents/graph/state.py`

当前已经定义了一版统一共享状态 `SupportAgentState`，主要包含：

- `user_query`
- `session_id`
- `intent`
- `route_reason`
- `retrieval`
- `action`
- `answer`
- `memory`
- `error`

同时提供了：

- `create_initial_state()`

作为统一入口状态构造函数。

### 2. orchestrator_agent

已新增：

- `supportAgents/agents/orchestrator_agent.py`

当前还是第一版规则路由，不是 LLM 路由。

已支持的意图类型：

- `doc_qa`
- `code_qa`
- `tool_only`
- `direct_answer`
- `fallback`

当前结论是：

- 第一版可以先规则路由跑通
- 后续更推荐升级成 LLM router

### 3. retrieval_agent

已新增：

- `supportAgents/agents/retrieval_agent.py`

当前不是自己重新实现检索，而是封装复用现有 `rag/` 能力，主要做了：

- 调用 `build_query_rewrite_result`
- 调用 `retrieve`
- 把结果转换成 graph 可传递的结构
- 统一生成 `context_text`

这一步的目标已经明确：

- `supportAgents` 负责 agent 封装
- `rag/` 继续作为现有检索能力底座

### 4. answer_agent

已新增：

- `supportAgents/agents/answer_agent.py`

当前 `answer_agent` 已经接入新的 `llm_clients`，默认可直接使用 DeepSeek。

它会根据不同意图：

- 选择不同系统提示
- 构造不同用户提示
- 最终生成回答并写回 `state["answer"]`

### 5. prompts 模块

已新增：

- `supportAgents/agents/prompts.py`

当前已经把 prompt 从业务逻辑里拆出来，集中管理，避免散落在 agent 内部。

已定义的 prompt 包括：

- `ORCHESTRATOR_SYSTEM_PROMPT`
- `ANSWER_SYSTEM_PROMPT`
- `TOOL_ONLY_SYSTEM_PROMPT`
- `FALLBACK_SYSTEM_PROMPT`
- `MEMORY_SYSTEM_PROMPT`

其中当前最重要的升级是：

- `answer_agent` 的角色已经从模糊的“企业支持助手”
- 收紧成“企业内部技术支持 Copilot”

并明确要求：

- 先给结论
- 再给依据
- 必要时给建议
- 不足就明确说不足
- 不伪造工具结果、日志、线上状态

### 6. graph builder

已新增：

- `supportAgents/graph/builder.py`

当前已经实现最小图编排：

- `orchestrator`
- `retrieval`
- `answer`

已经可以通过：

- `build_support_graph()`
- `run_support_graph()`

运行第一条最小闭环。

同时，为了避免循环导入，还调整了：

- `supportAgents/graph/__init__.py`

改成懒加载导出。

## 六、当前已验证过的内容

本阶段已经验证过：

- `DeepSeekClient` 可以正常实例化
- `create_llm_client("deepseek", "deepseek-v4-flash")` 可用
- `create_initial_state()` 可用
- `run_orchestrator()` 可用

- prompt 模块拆出后可以正常导入

当前还没有做的验证：

- 没有接入真实 `action_agent`
- 没有接入真实 `memory_agent`
- 没有把 `orchestrator` 升级成 LLM 路由

## 七、当前 prompt 设计结论

已经明确的一点是：

原本那种“企业支持助手”的提示词太泛，不适合作为正式版本。

当前结论是：

- prompt 要更窄
- 要更像企业内部技术支持工程师
- 要显式写清边界、输出结构和禁止项

当前推荐的 prompt 设计策略是：

- `orchestrator_agent` 只做路由判断
- `answer_agent` 负责最终用户可见回答
- `memory_agent` 负责沉淀可复用经验

## 八、当前最合理的下一步

结合当前代码状态，后续最合理的推进顺序建议如下：

1. 把 `orchestrator_agent` 从规则路由升级成 LLM 路由
2. 新增 `memory_agent`，把 `MEMORY_SYSTEM_PROMPT` 真正接起来
3. 新增 `action_agent`，后续把工具执行链路接入
4. 再把 graph 从最小闭环扩成完整五段流

如果优先考虑“更像真正的 multi-agent”，建议先做第 1 步。  
如果优先考虑“形成第二条业务闭环”，建议先做第 2 步。

## 九、额外注意事项

当前有两个工程层面的注意点：

### 1. 部分中文注释显示乱码

终端读取文件时，多处中文注释出现乱码。

当前判断更像是：

- 文件编码问题
- 或 PowerShell 输出编码问题

源码本身不一定损坏，但后续需要统一确认编码策略。

### 2. pyproject 仍有历史遗留命名

`pyproject.toml` 里仍然能看到 `tradingagents` 相关命名，和当前 `support-copilot` 项目不一致。

这更像历史模板遗留，后面建议统一收口。
