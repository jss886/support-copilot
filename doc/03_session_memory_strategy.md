# 会话状态与记忆策略

## 为什么需要会话状态

用户在支持场景中经常连续追问。如果系统不能保留前文，就会丢失问题背景。例如用户先问订单接口超时，再问是否和库存服务有关，这第二个问题必须结合前文才能回答。

因此项目设计中包含 `sessions` 表，用于保存用户当前会话的上下文状态。

## sessions 表建议保存的内容

- 当前主题
- 当前模块
- 最近一次用户问题
- 最近一次回答摘要
- Agent 状态信息

这些信息可以帮助 routing_agent 和 answer_agent 判断当前问题是否是追问，以及是否需要优先使用最近上下文。

## task_memory 的定位

`task_memory` 不保存普通聊天记录，而是保存可复用经验。只有当一次问答满足以下特征时，才建议写入：

1. 结论明确
2. 未来可能复现
3. 包含排查路径或经验判断
4. 可以抽象成规则

例如：

“遇到订单创建接口超时，优先检查库存服务是否发生线程池耗尽，再查看 Redis 热 key 是否导致库存扣减接口阻塞。”

这类结论就比普通对话更适合进入记忆库。

## 记忆写入时机

建议在回答生成后，由 `memory_agent` 负责判断是否值得沉淀，再统一写库。不要让主问答链路直接写入记忆，否则容易把很多低质量内容塞进 `task_memory`。

## 检索优先级建议

当用户问题与近期会话强相关时，应优先参考 `sessions` 中的上下文；当用户问题明显是在问可复用经验时，应优先检索 `task_memory` 和 `incident_cases`。
