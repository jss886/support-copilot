# 故障案例：订单创建接口超时

## 现象

用户反馈订单创建接口在高峰期频繁超时，接口平均响应时间从 300ms 升高到 8 秒以上。前端偶尔出现“系统繁忙，请稍后再试”的提示。

## 初步判断

订单创建涉及库存校验、优惠计算、用户画像查询和异步消息发送，链路较长。历史经验表明，这类问题优先检查库存服务与 Redis。

## 排查路径

1. 查看订单服务线程池与 Tomcat 工作线程是否打满。
2. 查看库存服务的 RT 和错误率。
3. 检查 Redis 是否出现热 key，特别是热门商品库存扣减场景。
4. 确认数据库是否存在锁等待，尤其是订单表和库存表。
5. 检查 MQ 发送是否阻塞主链路。

## 根因

最终确认是库存服务在促销期间出现线程池耗尽，导致库存扣减请求积压。同时某热门商品的库存 key 在 Redis 中请求过于集中，进一步放大了接口 RT。

## 解决方案

1. 扩大库存服务核心线程池和队列容量。
2. 对热点商品库存查询增加本地缓存与限流。
3. 将部分非关键逻辑异步化，避免阻塞订单主流程。
4. 对库存扣减链路增加熔断与降级策略。

## 可复用经验

如果订单创建接口在高峰期超时，优先排查库存服务线程池、Redis 热 key 和数据库锁等待。这三个方向通常比前端或网关更可能成为根因。
